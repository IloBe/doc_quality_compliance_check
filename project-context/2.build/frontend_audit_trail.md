# Frontend Page Documentation — Audit Trail

**Page label:** Compliance / Governance Evidence  
**Route:** `/audit-trail`  
**Protection:** Protected route inside `AppShell`  
**Owner persona:** `@frontend-eng`  
**Status:** Implemented, demo-by-default with optional PostgreSQL backend via env flag

## Purpose

Provide an immutable append-only governance timeline showing who changed what, when, and under which trace context. Serves as evidence chronology for review boards, auditors, and post-incident forensics — not a performance telemetry screen.

## Current implementation

- `frontend/pages/audit-trail.tsx` renders the full page.
- Renders standardized header via `PageHeaderWithWhy` (`Compliance / Governance Evidence` → `Audit Trail`).
- Renders `AuditTrailKpiGrid` with event count summaries.
- Renders `AuditTrailSchedulePanel` for upcoming internal/external audit date management.
- Renders `AuditTrailFiltersPanel` for substring search across event fields.
- Renders a two-column layout (xl:grid-cols-3):
  - **Left (2/3):** `AuditTrailTimelineTable` — event list with selection and audit-info toggle.
  - **Right (1/3):** `AuditTrailEventDetailsPanel` — details for the selected event.
- Uses bottom `Governance note` footer card (emerald).

## Data sources and state

### Data mode switch

Controlled by env var:

```bash
NEXT_PUBLIC_AUDIT_TRAIL_SOURCE=backend
```

- **Demo mode (default):** events generated by `createMockAuditTrailEvents()` from `lib/auditTrailViewModel`. No backend calls made.
- **Backend mode:** events fetched via `fetchAuditTrailEvents({ windowHours, limit: 250 })` from `lib/auditTrailClient`. Schedule loaded via `fetchAuditTrailSchedule()`.

### Event resolution

`getEffectiveAuditEvents()` from `auditTrailViewModel` selects the correct event list based on mode and applies window filtering.

### Event detail loading (backend mode only)

On event selection: `fetchAuditTrailEventDetail(eventId)` is called asynchronously; `selectedEventDetail` renders in the details panel.

### Search filter

`useSubstringFilter` hook with search text built from `event_time + event_type + actor_type:actor_id + subject_type:subject_id`.

## Audit schedule

- **Internal audit date:** ISO date string, editable, shown with days-until countdown.
- **External audit date + notified body:** editable pair (e.g., TÜV SÜD).
- `nearestFutureAudit` is derived from both dates via `getNearestFutureAudit()`.
- Save: `upsertAuditTrailSchedule()` in backend mode; demo mode shows inline update without API call.
- Schedule saving state: `isScheduleSaving` + `scheduleError` shown in the panel.

## UX and behavior contract

- **Demo banner:** blue info strip shown when not in backend mode; instructs user to set env var.
- **Read-only advisory:** amber-tinted info strip below banner confirms this is a read-only view; action surface is Auditor Workstation.
- **Window selector:** time-window buttons (7d, 30d, 90d, etc.) in header right area; changes reload event list.
- **Refresh button:** header right area; re-invokes `load()` to refresh from current source.
- **Open Auditor Workstation button:** header right area; navigates to `/auditor-workstation` for approve/reject actions.
- Event rows are selectable; selecting opens detail panel. Audit-info toggle per row available.
- Loading and error states render inline replacement panels.

## Known boundaries

- Read-only page — no write actions except schedule save.
- Detail panel populated from in-list data in demo mode; backend mode fetches richer detail separately.
- Actor, subject, and trace context visibility depends on what the backend emits in each event record.

## Acceptance criteria

- Demo mode loads and displays mock events by default without any env var.
- Backend mode loads real events and schedule when `NEXT_PUBLIC_AUDIT_TRAIL_SOURCE=backend`.
- Schedule save persists to backend in backend mode; shows demo note in demo mode.
- Window selector changes reload the event list.
- Event selection populates the detail panel.
- Substring filter reduces visible events in real time.
- Navigation to Auditor Workstation is reachable from the header.
